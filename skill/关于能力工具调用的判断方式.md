# 关于能力工具调用的判断方式

> 面向 AgentKernel 接入开发者。本文专门说明：当模型调用能力工具后，前端/业务端应该看哪个返回数据包、依赖哪些字段来判断当前是否还在等待，以及遇到页面刷新、用户中断交互这类场景时应该如何恢复。
>
> 本文不重复协议全字段定义；详细结构请配合阅读 [API.md](./API.md)、[TOOL_PROTOCOL.md](./TOOL_PROTOCOL.md)、[INTEGRATION-GUIDE.md](./INTEGRATION-GUIDE.md)。

---

## 一句话结论

**判断"能力是否还在等待"时，不要只看 `context.preview`，而要同时区分：**

1. **运行态**：当前 `run` 是否还活着，使用 `runtime.sessions` 和事件流判断；
2. **调用态**：某次能力调用是否已闭环，使用 `tool.call.request.payload.call_id` 与 `tool.execute.result.payload.call_id` 判断；
3. **历史态**：刷新后没有实时事件时，使用 `session.messages` 或 `context.preview` 中的 `tool_use` / `tool_result` 结构恢复。

---

## 核心原则

### 1. 不同数据包负责不同问题

| 你要判断的问题 | 优先使用的数据包 / 事件 | 关键字段 |
|------|------|------|
| 当前这个 session 是否还有活跃推理 | `runtime.sessions` | `payload.runs[].run_id`、`payload.runs[].status` |
| 模型是否刚发起了一次能力调用 | `tool.call.request` 事件 | `session_id`、`run_id`、`payload.call_id`、`payload.tool_name`、`payload.input` |
| 某次能力调用是否已经回填结果 | `tool.execute.result` 回填命令 / `tool.call.result` 事件 | `payload.call_id` |
| 页面刷新后如何还原工具链 | `session.messages` 或 `context.preview` | `message.run_id`、`content[].type`、`tool_use.id`、`tool_result.tool_use_id` |
| 当前模型实际能看到哪些消息 | `context.preview` | `payload.messages`、`payload.active_context` |

### 2. `context.preview` 不是运行态真相源

`context.preview` 回答的是：

- 当前活跃上下文的裁剪规则是什么；
- 当前有哪些消息会进入模型输入；
- 当前上下文消息数、seed 数、token 估算是多少。

`context.preview` **不直接回答**：

- 当前 `run` 是否仍在运行；
- 当前模型是否正在等待某个能力结果；
- 某个工具调用是否已经真正闭环。

所以，不能用下面这些字段判断能力等待态：

- `payload.active_context.context_id`
- `payload.active_context.mode`
- `payload.counts.active_messages`
- `payload.stats.message_count`

这些字段只能帮助你理解“上下文长什么样”，不能作为“能力是否 pending”的唯一依据。

## 运行态怎么判断

### 看哪个返回包

使用：

```text
runtime.sessions
```

它返回当前运行中的 session 和 run 列表。

### 看哪些字段

重点看：

- `payload.runs[]`
- `payload.runs[].session_id`
- `payload.runs[].run_id`
- `payload.runs[].status`

典型返回：

```json
{
  "success": true,
  "payload": {
    "running_session_count": 2,
    "running_run_count": 3,
    "sessions": ["session_a", "session_b"],
    "runs": [
      { "session_id": "session_a", "run_id": "run_xxx", "duration_ms": 1832, "status": "running" },
      { "session_id": "session_b", "run_id": "run_yyy", "duration_ms": 91234, "status": "cancelling" }
    ]
  }
}
```

### 应该怎么理解

- `status = "running"`：这轮推理还活着，可能正在输出文本，也可能正在等待工具结果；
- `status = "cancelling"`：已经请求中断，但还没彻底结束；
- 如果某个 `run_id` 不在 `payload.runs[]` 里，通常说明它已经结束、失败或被取消完成。

### 适合用来做什么

- 页面初始化时恢复“当前是否还在生成”；
- 刷新后判断旧的 pending 能力调用是否还有意义；
- 用户再次发消息前，判断是否需要先提示“上一个 run 还没结束”。

## 调用态怎么判断

### 真正的能力调用闭环

一轮标准工具调用链路如下：

```text
tool.call.request
  ↓
业务端执行能力 / 等待用户交互
  ↓
tool.execute.result
  ↓
tool.call.result
```

### 看哪个字段最关键

最关键的是：

- `tool.call.request.payload.call_id`

它是这一次能力调用的唯一 ID，也是回填结果时必须原样带回的主键。

### 必须一起记录的字段

业务端收到 `tool.call.request` 时，建议至少持久化：

- `session_id`
- `run_id`
- `payload.call_id`
- `payload.tool_name`
- `payload.input`
- `payload.timeout_ms`
- 本地状态字段，例如 `pending` / `awaiting_user` / `done` / `error` / `cancelled`

### 为什么只看 `run_id` 不够

一个 `run_id` 内可能连续发起多次工具调用。

所以：

- `run_id` 负责标识“这是一轮什么对话”；
- `call_id` 负责标识“这一轮里具体是哪一次能力调用”。

真正的 pending 判断必须落到 `call_id`。

### 判断规则

收到：

```json
{
  "type": "event",
  "event_type": "tool.call.request",
  "session_id": "my_session",
  "run_id": "run_xxx",
  "payload": {
    "call_id": "call_xxx",
    "tool_name": "AskUserQuestion",
    "input": {
      "question": "我可以帮你做这些事。你最想先了解或使用哪一类能力？",
      "options": ["图片理解/截图分析", "代码编写与调试", "文档写作/总结翻译"]
    }
  }
}
```

时，应立即创建一条 pending 记录。

只有在你发送：

```json
{
  "command": "tool.execute.result",
  "session_id": "my_session",
  "payload": {
    "call_id": "call_xxx",
    "result": "用户选择：代码编写与调试",
    "is_error": false
  }
}
```

之后，这次能力调用才算闭环。

### 推荐状态机

```text
tool.call.request 到达
↓
pending
↓
如果需要用户操作
awaiting_user
↓
用户完成交互
submitting_result
↓
已发送 tool.execute.result
done
```

如果失败或超时：

```text
awaiting_user
↓
timeout / user_abort / page_closed_long_time
↓
error
↓
发送 tool.execute.result(is_error=true)
```

## 刷新后怎么恢复

### 分两种来源恢复

页面刷新后，实时事件流会断掉，所以要同时依赖两类信息：

1. **本地或服务端持久化的 pending 调用记录**
2. **历史消息或上下文中的 `tool_use` / `tool_result` 结构**

最稳的做法是：

- 实时阶段以 `tool.call.request` 为真相源写入 pending；
- 刷新恢复阶段先读取 pending 存储；
- 再使用 `runtime.sessions` 和消息历史做校验；
- 最后决定是恢复 UI、自动补提问、还是清理失效记录。

### 刷新恢复的标准步骤

#### 第一步：读取 pending 记录

先读取你自己存储的 pending tool calls，例如：

```json
{
  "session_id": "session_6khf1n",
  "run_id": "run_xxx",
  "call_id": "call_uZFxycEPoGGRoNYVMsn4cdGq",
  "tool_name": "AskUserQuestion",
  "input": {
    "question": "我可以帮你做这些事。你最想先了解或使用哪一类能力？",
    "options": ["图片理解/截图分析", "代码编写与调试", "文档写作/总结翻译"]
  },
  "status": "awaiting_user"
}
```

#### 第二步：调用 `runtime.sessions`

确认这个 `run_id` 是否还在运行：

- 如果还在 `running`，说明恢复有意义；
- 如果已经不在运行列表里，这条 pending 记录大概率已经失效，需要继续做历史校验。

#### 第三步：拉取 `session.messages` 或 `context.preview`

检查该 `run_id` 下是否已经出现匹配结果：

- `assistant` 消息内出现 `content[].type = "tool_use"`，表示模型发起过调用；
- 后续如果有 `content[].type = "tool_result"`，并且 `tool_result.tool_use_id == tool_use.id`，表示结果已写回；
- 如果只有 `tool_use` 没有对应 `tool_result`，说明这次调用仍然处于未闭环状态。

#### 第四步：决定恢复策略

- `run` 仍在运行，且没有匹配 `tool_result`：恢复交互 UI；
- `run` 已结束，但历史里已经有 `tool_result`：清理本地 pending；
- `run` 已结束，历史里也没有 `tool_result`：视为异常中断，给出“重新提问”或“重试 run”的提示；
- `run` 在 `cancelling`：禁用交互，等待取消完成后再清理。

## 页面刷新时，应该看哪个返回包

### 场景 1：想知道系统现在还有没有活跃推理

看：

```text
runtime.sessions
```

字段：

- `payload.runs[].run_id`
- `payload.runs[].status`

### 场景 2：想知道某次能力调用最初是谁发起的、参数是什么

看：

```text
tool.call.request
```

字段：

- `session_id`
- `run_id`
- `payload.call_id`
- `payload.tool_name`
- `payload.input`

注意：这一包通常是事件流实时收到的，刷新后不会自动重放，所以你必须自己持久化。

### 场景 3：想知道工具结果是否已经写回模型

优先看：

```text
tool.call.result
```

字段：

- `payload.call_id`
- `payload.result`
- `payload.is_error`

但刷新后你未必能拿到这个事件，所以要再看历史消息中的 `tool_result`。

### 场景 4：想知道当前上下文视图里是否还能看到这段调用历史

看：

```text
context.preview
```

字段：

- `payload.messages`
- `payload.active_context.rules`
- `payload.counts`

注意：这只能帮助你恢复“当前模型可见的上下文消息”，不能代替 pending 状态存储。

### 场景 5：想从历史里兜底推断工具调用是否完成

看：

```text
session.messages
```

或：

```text
context.preview
```

字段：

- `message.run_id`
- `message.role`
- `content[].type`
- `tool_use.id`
- `tool_result.tool_use_id`
- `tool_result.is_error`

## 用户交互型能力怎么恢复

### 什么叫用户交互型能力

这类能力不是“后端立刻执行完再回填”，而是会停下来等待用户操作，例如：

- `AskUserQuestion`
- 文件选择器
- 授权确认
- 多选项决策
- 截图标注确认
- 设备选择 / 账号选择

这类能力的共同特点是：

- AgentKernel 已经发出 `tool.call.request`；
- 但业务端无法立即得到结果；
- 中间可能跨越用户等待、页面刷新、甚至前端重新连接。

### 恢复原则

**这类能力不能只依赖内存状态，必须持久化 pending call。**

推荐最小持久化字段：

| 字段 | 作用 |
|------|------|
| `session_id` | 归属会话 |
| `run_id` | 归属哪一轮推理 |
| `call_id` | 这次调用的唯一主键 |
| `tool_name` | 恢复时知道渲染哪类交互组件 |
| `input` | 恢复问题内容、选项、默认值等 |
| `status` | `awaiting_user` / `done` / `error` / `cancelled` |
| `created_at` | 超时控制 |
| `updated_at` | 状态追踪 |

### 恢复流程

```text
页面启动
↓
加载 pending user-interaction calls
↓
调用 runtime.sessions
↓
拉取 session.messages 或 context.preview
↓
逐条校验每个 pending call 是否已闭环
↓
恢复 UI 或清理无效记录
```

### `AskUserQuestion` 的推荐做法

收到：

```json
{
  "type": "event",
  "event_type": "tool.call.request",
  "session_id": "session_xxx",
  "run_id": "run_xxx",
  "payload": {
    "call_id": "call_xxx",
    "tool_name": "AskUserQuestion",
    "input": {
      "question": "请选择一个方向",
      "options": ["A", "B", "C"]
    }
  }
}
```

时：

1. 立刻保存 pending 记录；
2. 立刻展示提问 UI；
3. 如果用户刷新页面，重新从 pending 记录渲染该问题；
4. 用户完成选择后，发送 `tool.execute.result`；
5. 发送成功后，把本地状态更新为 `done`；
6. 等后续消息历史确认已经出现 `tool_result` 后，再彻底清理。

### 为什么不要只靠 `request_id`

`request_id` 只用于某一次命令请求和响应的关联，例如：

- 你调用 `context.preview` 的 `request_id = r12`
- 你调用 `runtime.sessions` 的 `request_id = r18`

它不代表某次工具调用本身，也不适合作为刷新恢复主键。

真正和工具调用绑定的是：

- `run_id`
- `call_id`

其中以 `call_id` 最关键。

## 如何从消息历史兜底推断 pending

### 可用字段

消息历史或上下文预览中，重点看：

- `message.run_id`
- `message.role`
- `content[].type`
- `tool_use.id`
- `tool_result.tool_use_id`
- `tool_result.is_error`

### 推断规则

在同一个 `run_id` 内：

1. 遇到 `assistant` 消息里的 `tool_use`，创建一个待匹配调用；
2. 遇到 `tool_result`，按 `tool_result.tool_use_id` 回挂；
3. 扫描完成后，仍然没有匹配结果的 `tool_use`，视为 pending；
4. 如果当前 `run_id` 还在 `runtime.sessions` 的运行列表中，则认为 pending 仍有效；
5. 如果当前 `run_id` 已不在运行列表中，则视为异常中断或历史残留，需要人工恢复或清理。

### 注意不要犯的错误

- 不要只看最后一条消息；
- 不要只靠消息顺序匹配工具结果；
- 不要只按工具名匹配；
- 不要把 `context.preview` 的消息数变化当成能力是否完成的依据；
- 不要把前端内存中的等待弹窗当成唯一状态源。

## 推荐实现方案

### 实时阶段

收到 `tool.call.request` 时：

1. 写入本地或服务端 pending store；
2. 以 `call_id` 作为幂等键；
3. 按 `tool_name` 路由到不同 UI 组件；
4. 等待用户操作或本地执行结束；
5. 回填 `tool.execute.result`。

### 刷新恢复阶段

页面初始化时：

1. 拉取 pending store；
2. 拉取 `runtime.sessions`；
3. 拉取 `session.messages` 或 `context.preview`；
4. 做三方比对：`pending store` + `running runs` + `tool_use/tool_result`；
5. 恢复仍有效的交互卡片；
6. 清理已经闭环或已失效的记录。

### 清理策略

- 已有 `tool_result`：清理 pending；
- `run` 已结束且超过超时时间：清理 pending，并标记为异常中断；
- 用户主动取消：发送错误结果或走 `run.cancel`，然后清理 pending；
- 重复收到同一个 `call_id`：按幂等规则返回旧结果，不重新创建新卡片。

---

## 推荐判断顺序

实际代码里，建议统一按下面顺序判断：

```text
1. 先看本地/服务端是否有 pending call_id
2. 再看 runtime.sessions 里 run_id 是否还活着
3. 再看 session.messages / context.preview 中是否已出现匹配 tool_result
4. 最后决定恢复 UI、继续等待、自动报错、还是清理记录
```

换句话说：

- **判断 run 是否存活**：看 `runtime.sessions`
- **判断调用是否闭环**：看 `call_id`
- **刷新后兜底恢复**：看 `tool_use.id` / `tool_result.tool_use_id`
- **判断当前模型可见上下文**：看 `context.preview`

---

## 最后总结

### 你真正需要记住的字段

最重要的只有这几组：

- 运行态：`runtime.sessions.payload.runs[].run_id`、`status`
- 调用态：`tool.call.request.payload.call_id`
- 历史关联：`tool_use.id`、`tool_result.tool_use_id`
- 聚合边界：`message.run_id`

### 最容易误用的字段

不要把下面这些当成 pending 判断主键：

- `request_id`
- `context_id`
- `active_context.mode`
- `counts.active_messages`

### 对用户交互型能力的最终建议

像 `AskUserQuestion` 这种需要用户参与的能力，必须实现：

1. `tool.call.request` 到来即持久化；
2. 用 `call_id` 恢复交互状态；
3. 页面刷新后先查 pending，再查 `runtime.sessions`，再查消息历史；
4. 用 `tool.execute.result` 完成闭环；
5. 对长时间无响应场景实现超时或取消策略，避免内核无限等待。
