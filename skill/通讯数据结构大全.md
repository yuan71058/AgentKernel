# AgentKernel 通讯数据结构大全

## 1. 文档目的

这份文档用于统一说明 AgentKernel 当前 WebSocket 协议的：

- 传输总结构
- 命令请求结构
- 响应返回结构
- 事件推送结构
- 字段含义与作用
- 当前实现状态

说明原则：

- 以当前代码实现为准，不按理想设计臆测
- 区分“已实现可用”和“已定义但暂未完整启用”
- 既描述字段结构，也描述实际作用

阅读建议：

- 如果只是第一次接入，先看 `skill/SKILL.md` 里的最小接入流程
- 如果要确认完整 WS 命令、响应、事件与字段，再看这份大全

## 2. 协议总原则

AgentKernel 的 WS 协议遵循以下原则：

- 所有操作都是 `Command`，由客户端发给服务端
- 所有状态变化都是 `Event`，由服务端主动推给客户端
- 所有命令结果都是 `Response`
- 所有流式运行时信号可以通过 `Event` 或 `Stream` 表达
- `session_id` 是最核心的业务作用域
- `run_id` 表示一次完整推理流程
- `trace_id` 预留给链路追踪

特别注意：

- 虽然 Rust 中定义了 `WsMessage::Command`，但当前服务端接收命令时并不是靠 `type = "command"` 来反序列化，而是直接按 JSON 字段读取 `command`、`request_id`、`session_id`、`payload`
- 所以客户端发命令时，不需要显式传 `type`

## 3. WS 顶层消息结构

### 3.1 客户端 -> 服务端：Command

```json
{
  "command": "session.send",
  "request_id": "r_001",
  "session_id": "debug_session",
  "payload": {
    "message": "你好"
  }
}
```

字段说明：

| 字段 | 类型 | 必填 | 作用 |
| --- | --- | --- | --- |
| `command` | `string` | 是 | 命令名，如 `session.send` |
| `request_id` | `string` | 是 | 请求唯一标识，用于和响应对应 |
| `session_id` | `string` | 通常必填 | 当前会话标识；系统级命令可为空 |
| `payload` | `object / null` | 否 | 命令参数体 |

作用：

- 这是所有客户端控制 Kernel 的统一入口
- 前端、脚本、业务服务端都通过它驱动 Runtime

---

### 3.2 服务端 -> 客户端：Response

```json
{
  "type": "response",
  "request_id": "r_001",
  "success": true,
  "payload": {
    "session_id": "debug_session",
    "content": "你好，有什么可以帮你？"
  }
}
```

失败响应示例：

```json
{
  "type": "response",
  "request_id": "r_001",
  "success": false,
  "payload": {
    "error": "upstream timeout",
    "source": "provider",
    "stage": "model.stream",
    "retryable": true,
    "message": "upstream timeout",
    "raw_error": "upstream timeout"
  }
}
```

字段说明：

| 字段 | 类型 | 必填 | 作用 |
| --- | --- | --- | --- |
| `type` | `string` | 是 | 固定为 `response` |
| `request_id` | `string` | 是 | 对应发起命令时的 `request_id` |
| `success` | `bool` | 是 | 是否成功 |
| `payload` | `object` | 是 | 返回数据；失败时至少包含 `error`，运行时错误还可能带 `source/stage/retryable/message/raw_error` |

作用：

- 表示某条命令已经被处理
- 是命令调用结果的同步确认面

---

### 3.3 服务端 -> 客户端：Event

```json
{
  "type": "event",
  "id": "evt_xxx",
  "event_type": "model.delta",
  "session_id": "debug_session",
  "run_id": "run_xxx",
  "trace_id": "",
  "timestamp": "2026-05-18T12:00:00Z",
  "payload": {
    "delta": "你好",
    "event_type": "text"
  },
  "event_seq": 12
}
```

字段说明：

| 字段 | 类型 | 必填 | 作用 |
| --- | --- | --- | --- |
| `type` | `string` | 是 | 固定为 `event` |
| `id` | `string` | 是 | 事件唯一 ID |
| `event_type` | `string` | 是 | 事件类型，如 `run.started`、`tool.call.request` |
| `session_id` | `string` | 是 | 事件所属会话 |
| `run_id` | `string` | 否 | 事件所属运行流程 |
| `trace_id` | `string` | 否 | 预留链路跟踪字段 |
| `timestamp` | `string(datetime)` | 是 | 事件时间 |
| `payload` | `object / null` | 是 | 事件内容 |
| `event_seq` | `u64` | 否 | Session 内单调递增事件序号，用于补拉 |

作用：

- 表示 Kernel 主动推送的运行时状态变化
- 用于 UI 渲染、调试、日志追踪、断线补拉

---

### 3.4 服务端 -> 客户端：Stream

```json
{
  "type": "stream",
  "session_id": "",
  "run_id": "",
  "event": "ping",
  "data": {
    "ok": true
  }
}
```

字段说明：

| 字段 | 类型 | 必填 | 作用 |
| --- | --- | --- | --- |
| `type` | `string` | 是 | 固定为 `stream` |
| `session_id` | `string` | 否 | 所属 session |
| `run_id` | `string` | 否 | 所属 run |
| `event` | `string` | 是 | 流式事件名 |
| `data` | `object / null` | 是 | 流式数据内容 |

当前实现状态：

- 当前 `Stream` 主要用于底层 `ping`
- 模型流式文本并不是通过 `type = "stream"` 推送，而是通过 `type = "event"` 且 `event_type = "model.delta"` 推送

## 4. 公共业务数据结构

### 4.1 Message

```json
{
  "message_id": "msg_xxx",
  "session_id": "debug_session",
  "run_id": "run_xxx",
  "role": "assistant",
  "kind": "normal",
  "content": [
    {
      "type": "text",
      "text": "你好"
    }
  ],
  "token_estimate": 12,
  "created_at": "2026-05-18T12:00:00Z",
  "metadata": {}
}
```

字段说明：

| 字段 | 类型 | 作用 |
| --- | --- | --- |
| `message_id` | `string` | 消息唯一 ID |
| `session_id` | `string` | 所属会话 |
| `run_id` | `string` | 产生该消息的运行 ID |
| `role` | `user/assistant/system/tool` | 消息角色 |
| `kind` | `normal/tool_result/compaction_summary/context_seed/system_note` | 消息语义类型 |
| `content` | `ContentBlock[]` | 消息内容块列表 |
| `token_estimate` | `u64` | 估算 token 数 |
| `created_at` | `datetime` | 创建时间 |
| `metadata` | `object` | 扩展信息 |

作用：

- 是全量历史保存的核心对象
- `session.messages`、上下文构建、持久化都围绕它展开

### 6.1 `session.send`

作用：

- 发起一次完整对话
- 自动创建 session（若不存在）
- 触发模型流式事件、工具调用事件、完成响应

请求：

```json
{
  "command": "session.send",
  "request_id": "r_send_1",
  "session_id": "debug_session",
  "payload": {
    "message": "帮我查询上海时间",
    "images": [],
    "max_rounds": 10
  }
}
```

请求字段：

| 字段 | 类型 | 必填 | 作用 |
| --- | --- | --- | --- |
| `message` | `string` | 是 | 用户输入文本 |
| `images` | `string[]` | 否 | 图像列表，当前按字符串数组传入 |
| `max_rounds` | `u32` | 否 | 最大工具轮次，默认 10 |

成功响应：

```json
{
  "type": "response",
  "request_id": "r_send_1",
  "success": true,
  "payload": {
    "session_id": "debug_session",
    "run_id": "run_xxx",
    "status": "completed",
    "partial_preserved": false,
    "content": "现在上海时间是 20:00。",
    "usage": {
      "input_tokens": 123,
      "output_tokens": 45
    },
    "traces": 1,
    "tool_calls_made": 1
  }
}
```

失败响应：

```json
{
  "type": "response",
  "request_id": "r_send_1",
  "success": false,
  "payload": {
    "error": "upstream timeout",
    "source": "provider",
    "stage": "model.stream",
    "retryable": true,
    "message": "upstream timeout",
    "raw_error": "upstream timeout"
  }
}
```

补充说明：

- 这条命令执行过程中，客户端通常还会同时收到若干 `event`
- 正常完成时通常会收到 `run.started`、`model.delta`、`tool.call.request`、`tool.call.result`、`model.completed`、`run.completed`
- 如果上层主动发送 `run.cancel`，则会在最终响应里看到 `status = "cancelled"`，并可能带 `partial_preserved = true`

### 6.2 `tool.register`

作用：

- 向 Kernel 注册一个可被模型调用的工具
- 若带 `session_id`，会把当前工具快照持久化进该 session metadata

请求：

```json
{
  "command": "tool.register",
  "request_id": "r_tool_reg_1",
  "session_id": "debug_session",
  "payload": {
    "tool_name": "get_time",
    "description": "获取当前时间",
    "schema": {
      "type": "object",
      "properties": {
        "timezone": { "type": "string" }
      },
      "required": ["timezone"]
    },
    "client_id": "web_debug_console",
    "permissions": [],
    "timeout_ms": 30000,
    "tags": ["builtin"]
  }
}
```

请求字段：

| 字段 | 类型 | 必填 | 作用 |
| --- | --- | --- | --- |
| `tool_name` | `string` | 是 | 工具名 |
| `description` | `string` | 否 | 工具描述 |
| `schema` | `object` | 否 | JSON Schema |
| `client_id` | `string` | 否 | 注册来源客户端 |
| `permissions` | `string[]` | 否 | 权限标签 |
| `timeout_ms` | `u64` | 否 | 超时，默认 30000 |
| `tags` | `string[]` | 否 | 工具标签 |

成功响应：

```json
{
  "type": "response",
  "request_id": "r_tool_reg_1",
  "success": true,
  "payload": {
    "registered": "get_time",
    "session_id": "debug_session"
  }
}
```

额外事件：

- 会额外发出 `tool.registered`

### 6.6 `tool.execute.result`

作用：

- 这是客户端收到 `tool.call.request` 后，回传工具执行结果的专用命令
- 不属于常量公开命令，但已被服务端特殊处理

请求：

```json
{
  "command": "tool.execute.result",
  "request_id": "r_tool_exec_result_1",
  "session_id": "debug_session",
  "payload": {
    "call_id": "call_xxx",
    "result": "2026-05-18 20:00:00",
    "is_error": false
  }
}
```

请求字段：

| 字段 | 类型 | 必填 | 作用 |
| --- | --- | --- | --- |
| `call_id` | `string` | 是 | 对应 `tool.call.request` 的调用 ID |
| `result` | `string` | 是 | 执行结果文本 |
| `is_error` | `bool` | 否 | 是否为错误结果 |

当前实现特点：

- 服务端接收后会唤醒等待中的 `WsToolRouter`
- 这条命令当前没有显式 `send_ok` 响应
- 它更像内部控制回传，而不是常规 UI 查询命令

### 6.27 `run.cancel`

作用：

- 协议层用于请求中断指定 `run_id`
- 典型场景是用户点击“停止生成”，或业务端发现本轮输出需要被主动打断

请求：

```json
{
  "command": "run.cancel",
  "request_id": "r_run_cancel_1",
  "session_id": "debug_session",
  "payload": {
    "run_id": "run_xxx"
  }
}
```

请求字段：

| 字段 | 类型 | 必填 | 作用 |
| --- | --- | --- | --- |
| `run_id` | `string` | 是 | 要中断的运行 ID |

成功响应：

```json
{
  "type": "response",
  "request_id": "r_run_cancel_1",
  "success": true,
  "payload": {
    "status": "cancelling",
    "session_id": "debug_session",
    "run_id": "run_xxx",
    "cancelled": true
  }
}
```

未命中响应：

```json
{
  "type": "response",
  "request_id": "r_run_cancel_1",
  "success": true,
  "payload": {
    "status": "not_found",
    "session_id": "debug_session",
    "run_id": "run_xxx",
    "cancelled": false
  }
}
```

当前实现状态：

- 会按 `run_id` 查找当前运行中的任务
- 如果命中，会触发运行时取消标记，并停止当前模型流或等待中的工具回传
- 真正中断完成后，客户端会再收到 `run.cancelled` 事件，以及 `session.send` 的最终响应
- 如果模型在中断前已经输出了部分文本，Kernel 会尽量保留，并在最终响应中返回：

```json
{
  "session_id": "debug_session",
  "run_id": "run_xxx",
  "status": "cancelled",
  "partial_preserved": true,
  "content": "这是中断前已经生成的部分内容"
}
```

## 7. 事件结构大全

统一结构：

```json
{
  "type": "event",
  "id": "evt_xxx",
  "event_type": "事件名",
  "session_id": "debug_session",
  "run_id": "run_xxx",
  "trace_id": "",
  "timestamp": "2026-05-18T12:00:00Z",
  "payload": {},
  "event_seq": 1
}
```

---

### 7.1 已实际发出的事件

| 事件名 | 来源 | 作用 | 当前状态 |
| --- | --- | --- | --- |
| `session.created` | SessionManager | 创建或自动创建 session | 已发出 |
| `session.closed` | SessionManager | session 关闭或删除 | 已发出 |
| `run.started` | Runtime | run 开始 | 已发出 |
| `model.delta` | Runtime | 模型流式增量 | 已发出 |
| `model.completed` | Runtime | 模型完整输出完成 | 已发出 |
| `run.cancelled` | Runtime | run 被用户主动中断 | 已发出 |
| `run.completed` | Runtime | run 正常完成 | 已发出 |
| `tool.call.request` | WsToolRouter | 请求客户端执行工具 | 已发出 |
| `tool.call.result` | Runtime | 工具执行结果写回 | 已发出 |
| `tool.registered` | ToolManager / WS | 工具注册完成 | 已发出 |
| `context.threshold.reached` | Runtime | 上下文 token 达阈值 | 已发出 |
| `context.compaction.applied` | WS Context Handler | 应用压缩摘要 | 已发出 |
| `context.reset` | WS Context Handler | 上下文被重置 | 已发出 |
| `context.updated` | WS Context Handler | 上下文规则更新 | 已发出 |
| `context.seed.added` | WS Context Handler | 新增 seed | 已发出 |
| `error` | Runtime | 运行中错误与重试信息 | 已发出 |

### 7.3 事件逐项说明

#### `session.created`

作用：

- 表示 session 被创建
- 包括主动创建和 `get_or_create` 自动创建

典型 payload：

```json
{ "title": "xxx" }
```

或：

```json
{ "auto_created": true }
```

---

#### `session.closed`

作用：

- 表示 session 被关闭或删除

典型 payload：

```json
{ "reason": "deleted" }
```

也可能为空 payload。

---

#### `run.started`

作用：

- 一次模型推理开始

典型 payload：

```json
{
  "provider": "openai",
  "model": "gpt-4.1"
}
```

---

#### `run.cancelled`

作用：

- 表示某个 `run_id` 已被主动中断
- 如果中断前已经收到部分 `model.delta`，Kernel 会尽量把这部分文本保留下来

典型 payload：

```json
{
  "reason": "user_cancelled",
  "partial_content": "这是中断前已经生成的部分内容",
  "preserved": true,
  "duration_ms": 842
}
```

说明：

- `partial_content` 是中断时已保留的文本快照
- `preserved = true` 表示这段内容已经作为 assistant 消息保留下来
- 前端或业务端应把它理解为“本轮被打断，但中断前内容可继续展示”

---

#### `model.delta`

作用：

- 模型流式输出增量
- 前端流式渲染主要依赖它

典型 payload：

```json
{
  "delta": "你好",
  "event_type": "text"
}
```

注意：

- 这里 payload 里的 `event_type` 表示流式子类型，如 `text`、`thinking`
- 它不是外层 envelope 的 `event_type`

---

#### `model.completed`

作用：

- 表示模型本轮输出最终完成

典型 payload：

```json
{
  "content": "你好，有什么可以帮你？"
}
```

---

#### `run.completed`

作用：

- 表示某个 `run_id` 正常结束
- 它和 `model.completed` 的区别是：`model.completed` 更偏最终文本，`run.completed` 更偏运行统计

典型 payload：

```json
{
  "status": "completed",
  "input_tokens": 123,
  "output_tokens": 45,
  "total_tokens": 168,
  "tool_calls_made": 1,
  "duration_ms": 1200
}
```

---

#### `tool.call.request`

作用：

- 通知客户端去执行工具

典型 payload：

```json
{
  "tool_name": "get_time",
  "call_id": "call_xxx",
  "input": {
    "timezone": "Asia/Shanghai"
  }
}
```

客户端收到后通常应：

1. 本地查找对应工具实现
2. 执行工具
3. 调用 `tool.execute.result` 回传结果

---

#### `tool.call.result`

作用：

- 记录工具执行结果，便于前端调试、日志追踪、回放

典型 payload：

```json
{
  "tool_name": "get_time",
  "call_id": "call_xxx",
  "result": "2026-05-18 20:00:00",
  "is_error": false
}
```

注意：

- 这是“执行结果事件”
- 不是再次要求前端执行工具

---

#### `tool.registered`

作用：

- 通知工具已注册

典型 payload：

```json
{
  "tool_name": "get_time",
  "client_id": "web_debug_console",
  "session_id": "debug_session"
}
```

说明：

- 当前工具管理器和 WS 注册流程都可能发这个事件
- 不同路径 payload 细节可能略有差异

---

#### `context.threshold.reached`

作用：

- 当前上下文 token 占比达到压缩阈值
- 通知业务端可以考虑做 compaction

典型 payload：

```json
{
  "estimated_tokens": 100000,
  "window_tokens": 128000,
  "usage_percent": 78
}
```

---

#### `context.compaction.applied`

作用：

- 表示压缩摘要已被应用到上下文

典型 payload：

```json
{
  "context": {},
  "seed": {}
}
```

---

#### `context.reset`

作用：

- 表示上下文规则被重置为默认状态

典型 payload：

```json
{
  "context": {}
}
```

---

#### `context.updated`

作用：

- 表示上下文规则发生一般性更新

典型 payload：

```json
{
  "action": "exclude",
  "context": {}
}
```

可能的 `action`：

- `exclude`
- `include_after`
- `keep_recent`

---

#### `context.seed.added`

作用：

- 表示新增了一个 seed

典型 payload：

```json
{
  "seed": {}
}
```

---

#### `error`

作用：

- 表示运行期错误信息
- 当前 Core 不再内部自动重试，而是只上报错误属性给 WS 上层决定

典型 payload：

```json
{
  "error": "upstream timeout",
  "source": "provider",
  "stage": "model.stream",
  "retryable": true,
  "message": "upstream timeout",
  "raw_error": "upstream timeout"
}
```

## 9. 典型完整链路

### 9.1 普通对话

1. 客户端发送 `session.send`
2. 服务端返回多个 `event`
3. 客户端收到 `run.started`
4. 客户端持续收到 `model.delta`
5. 客户端收到 `model.completed`
6. 服务端最终返回本次命令的 `response`

### 9.2 带工具调用的对话

1. 客户端发送 `session.send`
2. 服务端发出 `run.started`
3. 模型产生工具调用意图
4. 服务端发出 `tool.call.request`
5. 客户端本地执行工具
6. 客户端发送 `tool.execute.result`
7. 服务端继续推进模型推理
8. 服务端发出 `tool.call.result`
9. 服务端继续发出 `model.delta`
10. 服务端发出 `model.completed`
11. 服务端返回最终 `response`

### 9.3 断线重连后的事件恢复

1. 客户端记录上次看到的 `event_seq`
2. 重连后调用 `events.pull`
3. 如需声明订阅，可再调用 `events.subscribe`
4. 将补拉回来的 `events` 按序合并进本地事件流

### 9.4 用户主动中断对话

1. 客户端先通过 `session.send` 发起本轮对话
2. 服务端开始推送 `run.started`、`model.delta`
3. 用户点击停止，客户端发送 `run.cancel`
4. 服务端先返回 `response.success = true`，其中 `payload.status = "cancelling"`
5. 真正中断完成后，服务端发出 `run.cancelled`
6. 如果已有部分输出，会在 `run.cancelled.payload.partial_content` 和最终 `session.send` 响应中体现
7. `session.send` 最终响应会返回 `status = "cancelled"`，并通过 `partial_preserved` 告诉上层这段内容是否已保留

## 10. 当前实现差异与注意事项

### 10.1 `events.subscribe` 只是部分完成

当前状态：

- 有命令
- 有确认响应
- 有基于 `since_seq` 的补发能力
- 但没有形成连接级常驻的完整事件订阅器

建议理解：

- 它目前更像“订阅协议语义已存在”
- 还不是严格意义上完整的 server-side live subscription manager

### 10.2 `tool.execute.result` 没有标准响应

当前状态：

- 服务端会消费这条命令并唤醒工具等待通道
- 但不会额外返回 `send_ok`

影响：

- 前端不应把它当作普通查询命令依赖响应渲染

### 10.3 `run.cancel` 是“请求中断”，不是同步完成通知

当前状态：

- `run.cancel` 的同步 `response` 只表示“已收到取消请求”
- 真正的结束结果要以 `run.cancelled` 事件和 `session.send` 最终响应为准
- 如果业务端要做 UI 停止按钮，推荐流程是：按钮发送 `run.cancel` -> UI 进入 `cancelling` -> 等待 `run.cancelled`

### 10.4 `provider.update.tools_mode` 结构有定义，但 handler 未真正写入

当前状态：

- 请求结构体定义里有 `tools_mode`
- 但当前更新逻辑未处理它

### 10.5 `session.messages` 返回的是简化视图

当前状态：

- 返回 `message_id / role / kind / text / created_at`
- 不是完整 `Message.content`

影响：

- 它适合列表展示
- 不适合做完整消息协议回放

### 10.6 `event_type` 字段不能写成 `type`

原因：

- 外层 `WsMessage` 已经使用 `type` 作为枚举 tag
- `EventEnvelope` 内部必须使用 `event_type`

## 11. 推荐前端接入规范

### 11.1 命令发送统一结构

建议前端统一封装：

```json
{
  "command": "xxx",
  "request_id": "唯一ID",
  "session_id": "当前session",
  "payload": {}
}
```

### 11.2 响应处理原则

- `response` 只处理命令结果
- `event` 只处理 Kernel 主动状态变化
- `stream` 只处理底层流信号

### 11.3 工具回调处理原则

- 收到 `tool.call.request` 才去执行本地工具
- 执行完后必须按 `call_id` 回传 `tool.execute.result`

### 11.4 断线恢复原则

- 本地保存最后一个 `event_seq`
- 重连后优先 `events.pull`

## 12. 一句话总结

AgentKernel 当前 WS 协议可以概括为：

- `Command` 负责控制
- `Response` 负责确认
- `Event` 负责状态变化
- `Stream` 负责少量底层流信号
- `session_id` 是业务域
- `run_id` 是单次推理域
- `event_seq` 是事件恢复域

- 每个命令的 JSON Schema
- 完整事件 payload 样例库
- 前后端时序图
- 已实现/规划中/废弃 字段状态表
