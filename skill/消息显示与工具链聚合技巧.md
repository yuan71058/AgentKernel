# 消息显示与工具链聚合技巧

> 面向 AgentKernel 接入开发者。本文专门说明前端/业务端如何展示 `session.messages` 与 `context.preview` 返回的消息，避免把 AI 工具链明细错误渲染成多段碎片对话。

---

## 一句话原则

**聊天区按 run 聚合，调试区按原始消息展开。**

用户看到的聊天流应该是：

```text
用户消息
助手完整回复
工具链卡片（可折叠）
```

而不是：

```text
助手片段
工具请求
工具结果
助手片段
工具请求
工具结果
助手片段
```

原始明细当然要保留，但应该放到“调试面板 / 全量历史 / trace 面板”，不要直接当聊天 UI。

---

## 为什么会出现碎片消息

一次用户输入对应一个 `run_id`。

在这个 run 内，模型可能经历多轮：

```text
assistant text
assistant tool_use
user tool_result
assistant text
assistant tool_use
user tool_result
assistant final text
```

这些都是同一轮 AI 运行的一部分。业务端如果直接按消息数组逐条展示，就会把一次完整回答拆成很多段。

正确做法是：

- 同一个 `run_id` 内的 assistant 文本合并成一条完整回复；
- 同一个 `run_id` 内的所有 `tool_use` 合并成一张工具链卡片；
- `tool_result` 通过 `tool_result.tool_use_id == tool_use.id` 挂回对应工具；
- 不要跨 `run_id` 聚合。

---

## 可用参照 ID

AgentKernel 已经提供聚合所需 ID：

| 字段 | 作用 |
|------|------|
| `message.run_id` | 判断消息属于哪一次用户请求 / AI run |
| `content[].type` | 判断内容块类型：`text` / `tool_use` / `tool_result` |
| `tool_use.id` | 工具调用唯一 ID |
| `tool_result.tool_use_id` | 回填结果时关联对应工具调用 |
| `message.role` | 消息角色：`user` / `assistant` / `tool` / `system` |

关键点：**聚合边界用 `run_id`，工具结果匹配用 `tool_use_id`。**

---

## session.messages 和 context.preview 都一样处理

`session.messages` 看的是历史分页。

`context.preview` 看的是当前模型实际可见上下文。

两者返回的消息结构基本一致，所以前端应该复用同一套渲染逻辑：

```text
原始 messages
↓
按 run_id 聚合
↓
按 tool_use_id 挂载结果
↓
输出聊天 UI 数据
```

不要给历史消息和 active context 写两套不同逻辑，否则很容易一个正常、一个碎片化。

---

## 推荐展示结构

### 聊天区

```text
用户：帮我打开 TikTok

助手：我已经打开设备并完成应用定位，正在进入目标页面。

工具链：共调用 7 次，全部完成
  - ocr_screen
    - 输入参数
    - 返回结果
  - tap
    - 输入参数
    - 返回结果
  - node_query
    - 输入参数
    - 返回结果
```

### 调试区

调试区可以展示完整原始消息：

```text
msg_1 user text
msg_2 assistant text + tool_use
msg_3 user tool_result
msg_4 assistant text + tool_use
...
```

这两个区的目的不同：

- 聊天区：给人看结果和过程摘要；
- 调试区：给开发者查协议细节。

---

## 聚合算法

伪代码：

```text
for message in messages:
  run_id = message.run_id

  if message.role == user 且不是 tool_result:
    输出普通用户消息

  if message.role == assistant:
    group = get_or_create_assistant_group(run_id)

    for text block:
      group.text += block.text

    for tool_use block:
      group.tool_calls[tool_use.id] = {
        id,
        name,
        input,
        status: pending
      }

  if message contains tool_result:
    找到同 run_id 内 tool_calls[tool_result.tool_use_id]
    写入 result / is_error / status
```

注意：assistant 后续继续输出文本时，不要创建新的助手消息，应该追加到同一个 `run_id` 的 group。

---

## 工具链卡片必须显示入参和返回

每个工具调用至少展示：

| 字段 | 来源 |
|------|------|
| 工具名 | `tool_use.name` |
| 调用 ID | `tool_use.id` |
| 入参 | `tool_use.input` |
| 返回 | 匹配到的 `tool_result.content` |
| 状态 | 是否匹配到 result、是否 `is_error` |

只显示返回、不显示入参是不够的。移动端自动化场景里，点击、OCR、节点查询都必须能看到入参，否则无法判断 AI 为什么点了某个坐标或查了某个节点。

## 常见错误

### 错误 1：逐条展示所有消息

结果：工具调用多时，聊天区像流水账，用户看不出最终回复。

正确：聊天区聚合，调试区展开。

---

### 错误 2：遇到 assistant text 就断开工具链

同一个 run 内，assistant 可能多次输出文本。不要因为中间出现 text 就把工具链拆成多段。

正确：同一 `run_id` 内持续合并。

---

### 错误 3：只按相邻顺序匹配工具结果

工具结果不应该靠“下一条消息”猜。

正确：必须用 `tool_result.tool_use_id == tool_use.id`。

---

### 错误 4：跨 run 聚合

不同用户输入对应不同 run，不能混在一起。

正确：`run_id` 不同就新建 group。

---

### 错误 5：只展示 tool result，不展示 tool input

这会导致移动端自动化无法排查。

正确：工具链卡片必须同时展示输入参数和返回结果。

---

## 最小实现建议

业务端可以把一轮 run 映射成这个 UI 数据结构：

```json
{
  "role": "assistant",
  "run_id": "run_xxx",
  "text": "完整助手输出文本...",
  "tool_calls": [
    {
      "id": "call_xxx",
      "name": "ocr_screen",
      "input": {},
      "result": "...",
      "status": "done",
      "is_error": false
    }
  ]
}
```

最终前端展示一条 assistant 消息 + 一张可折叠工具链卡片即可。

---

## 判断是否是工具结果消息

不要只看顶层 `role`。

当前工具结果通常可能表现为：

```json
{
  "role": "user",
  "content": [
    {
      "type": "tool_result",
      "tool_use_id": "call_xxx",
      "content": "..."
    }
  ]
}
```

所以判断工具结果要看：

```text
content[].type == "tool_result"
```

而不是只看：

```text
role == "tool"
```

---

## 结论

AgentKernel 的输出格式是可以支持正确聚合的：

- `run_id` 负责同一轮边界；
- `tool_use.id` / `tool_result.tool_use_id` 负责工具调用关联；
- `content[].type` 负责区分文本、工具请求、工具结果。

业务端不要原样把协议消息当聊天消息展示。协议消息是运行时日志，聊天消息需要经过一次 UI 聚合。
