# 压缩上下文技巧

> 面向 AgentKernel 接入开发者。本文只讲“怎么搭配命令做上下文压缩”，不重复完整字段说明；字段细节看 [API.md](./API.md)。

---

## 一句话原则

**压缩不是 Core 自动做摘要，而是业务端编排一个外部压缩工作流。**

Core 只负责三件事：

1. 提供历史消息读取能力：`session.messages`
2. 接收摘要注入：`context.seed.set`
3. 控制旧历史是否进入模型：`context.trim.set`

不要让 Core 猜“该总结哪一段、摘要是否可信、该从哪里继续”。这些都属于业务策略。

---

## 推荐流程：临时压缩 Session

当某个主 session 历史太长时，推荐业务端创建一个临时压缩 session，让另一个 AI 专门负责总结。

流程：

```text
主 session 过长
↓
业务端读取主 session 历史
↓
创建临时 compact session
↓
把待压缩历史发给 compact AI
↓
compact AI 输出结构化摘要
↓
业务端审核 / 校验摘要
↓
写入主 session 的 compaction_summary seed
↓
用 context.trim.set 把主 session 裁剪到旧历史末尾
↓
主 session 后续继续对话
```

---

## 命令搭配

### 1. 读取主 session 历史

```json
{
  "command": "session.messages",
  "session_id": "main_session",
  "payload": {
    "page": 0,
    "limit": 300
  }
}
```

业务端自行分页读取需要压缩的范围。不要只读当前 `context.preview`，因为 preview 只代表当前模型可见视图，不一定是完整历史。

---

### 2. 临时 session 生成摘要

可以用一个专门的 session，例如：

```text
main_session: jpy:controller:674
compact_session: compact:jpy:controller:674:001
```

给 compact session 配一个专门 prompt，要求它输出稳定结构，例如：

```text
你是上下文压缩器。请把输入历史压缩成可继续任务的摘要。
必须保留：用户目标、关键事实、已完成动作、工具结果结论、未完成事项、风险和下一步。
不要保留：冗余日志、重复截图描述、长工具原始返回。
```

然后通过 `session.send` 把待压缩历史交给 compact session。

---

### 3. 把摘要注入主 session

```json
{
  "command": "context.seed.set",
  "session_id": "main_session",
  "payload": {
    "kind": "compaction_summary",
    "content": "这里是压缩后的历史摘要...",
    "enabled": true,
    "priority": 100
  }
}
```

注意：`context.seed.set` **只写 seed**。

它不会：

- 移动 active context
- 修改 trim 策略
- 自动 include_after
- 删除历史消息

这点是故意设计的。Seed 是“摘要注入”，Trim 是“历史可见性控制”，不要把两个动作暗中绑定。

---

### 4. 显式裁剪旧历史

如果摘要已经覆盖了某段旧历史，就再显式调用：

```json
{
  "command": "context.trim.set",
  "session_id": "main_session",
  "payload": {
    "mode": "include_after",
    "message_id": "msg_last_compacted"
  }
}
```

`message_id` 建议取“已被摘要覆盖的最后一条消息”。

这样后续模型输入会变成：

```text
system prompt
+
compaction_summary seed
+
msg_last_compacted 之后的新消息
```

旧消息仍永久保存在历史中，只是不再进入当前模型上下文。

---

## 自动阶梯裁剪怎么配

如果暂时不做外部摘要，只想降低多轮历史膨胀，可以用：

```json
{
  "command": "context.trim.set",
  "session_id": "main_session",
  "payload": {
    "mode": "checkpoint",
    "trigger_max_context_messages": 300,
    "retain_recent_turns": 20
  }
}
```

语义：

- 用“上下文消息数”判断是否太长，工具消息也算；
- 用“用户对话轮次”决定保留范围，tool_result 不算用户轮次；
- 只在下一次 `session.send` 用户消息边界触发，不在 AI 工具链中途裁剪。

适合解决：多轮对话变长。

不适合解决：少数几轮里工具调用爆炸。

---

## 少轮但工具消息爆炸怎么办

比如只有 3 轮对话，但工具调用已经产生 300 条消息。不要指望 `checkpoint` 硬切。

这类问题应该用单独策略处理：

1. 工具结果不要直接塞长原文，优先返回摘要 + 文件引用；
2. 大型结果写到业务端存储，只把关键结论回填为 tool_result；
3. 单个重 run 结束后，由业务端做 run-level summary；
4. 再用 `context.seed.set(kind=compaction_summary)` 注入总结；
5. 最后用 `context.trim.set(mode=include_after)` 明确切掉已总结范围。

核心判断：

> 多轮历史过长，用 checkpoint；单轮工具链过重，用工具结果压缩 / run summary。

## 推荐摘要格式

建议 compact AI 输出 Markdown，结构稳定，便于后续模型使用：

```md
## 用户目标
- ...

## 已知事实
- ...

## 已完成动作
- ...

## 工具结果结论
- ...

## 当前状态
- ...

## 未完成事项
- ...

## 风险与注意
- ...

## 下一步建议
- ...
```

不要把原始日志、长 JSON、截图 OCR 全量塞进摘要。摘要的价值是“继续任务所需状态”，不是“历史备份”。

## 常见错误

### 错误 1：只写 seed，不 trim

结果：摘要进来了，但旧历史还在上下文里，token 没降，模型还可能被旧细节干扰。

正确：写完 `context.seed.set` 后，再显式 `context.trim.set(mode=include_after)`。

---

### 错误 2：让 seed.set 自动裁剪

不要这么设计。业务端必须显式裁剪，否则 seed 写入会有副作用，后续很难排查。

---

### 错误 3：裁剪到工具链中间

不要把 `message_id` 指到一组 `tool_use/tool_result` 中间。最好裁到一个完整任务段之后，至少保证裁剪点之前的工具链已经闭环。

---

### 错误 4：把 checkpoint 当压缩

`checkpoint` 只是“隐藏一部分历史消息”，不是摘要压缩。真正压缩一定需要外部 AI 生成 summary，再通过 seed 注入。

---

## 推荐组合

| 场景 | 推荐做法 |
|------|----------|
| 多轮普通聊天变长 | `context.trim.set(mode=checkpoint)` |
| 需要保留长期状态 | `context.seed.set(kind=compaction_summary)` + `context.trim.set(mode=include_after)` |
| 单轮工具结果巨大 | 工具结果摘要化 / 文件引用 / run summary |
| 想恢复全量上下文 | `context.trim.set(mode=none)` |
| 想排除污染片段 | `context.exclude` |

---

## 最小可用压缩工作流

```text
1. session.messages 读取待压缩历史
2. compact session 生成 summary
3. context.seed.set(kind=compaction_summary) 写入主 session
4. context.trim.set(mode=include_after, message_id=最后一条已压缩消息)
5. context.preview 确认可见上下文 = summary seed + 新消息
```

这套流程是最稳的：职责清楚、可回滚、可审计，也不会让 Core 变成业务编排器。
